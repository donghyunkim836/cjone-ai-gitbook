# 06. CLAUDE.md - 작업 기억

Claude Code는 한 번의 대화를 "세션"이라고 부릅니다. 세션을 종료하고 새로 시작하면 백지 상태입니다. 이전에 무슨 대화를 했는지, 어떤 규칙을 정했는지 기억하지 못합니다.

한 시간 동안 프로젝트 구조를 설명하고, 코딩 스타일을 맞추고, 보고서 형식을 정했는데 다음 세션에서 다시 처음부터. 이런 경험이 반복되면 AI를 쓰는 의미가 줄어듭니다.

CLAUDE.md는 이 문제를 해결합니다. 프로젝트 폴더에 두는 마크다운 파일로, Claude는 세션이 시작될 때 이 파일을 컨텍스트에 자동으로 주입합니다. 작업 규칙, 선호사항, 프로젝트 정보를 한 번만 적어두면 매 세션마다 반복할 필요가 없습니다.

## Step 1. /init으로 자동 생성

프로젝트 폴더에서 Claude Code를 실행하고 `/init`을 입력합니다.

```
/init
```

Claude가 프로젝트의 파일 구조, package.json, README 등을 분석하여 CLAUDE.md 초안을 생성합니다. 이미 CLAUDE.md가 있으면 덮어쓰지 않고 개선안을 제안합니다.

## Step 2. 직접 작성하기

자동 생성 대신 직접 내용을 지정하면 더 정확한 지침을 줄 수 있습니다.

**비즈니스 업무용 예시 (마케팅팀, 기획팀 등):**

```markdown
# 마케팅팀 업무 자동화

## 작업 규칙
- 응답은 한국어로 작성
- 보고서는 격식체(합니다/습니다) 사용
- 금액 표기: 원화 기호 없이 숫자만, 천 단위 쉼표 포함 (예: 1,200,000)
- 날짜 형식: YYYY-MM-DD (예: 2024-03-15)

## 자주 쓰는 작업
- 주간 보고서: 성과 요약 + 다음 주 계획 + 이슈 사항
- 이메일 분류: 긴급도(높음/중간/낮음) + 주제별 분류
- 경쟁사 분석: 제품, 가격, 강점/약점 비교표

## 참고 정보
- 팀장: 박팀장 (보고 대상)
- 주간 회의: 매주 월요일 오전 10시
- 보고서 마감: 매주 금요일 오후 5시
```

**개발 프로젝트용 예시:**

```markdown
# 프로젝트 개요
할일 관리 REST API - 사용자별 할일 CRUD 및 카테고리 분류

## 기술 스택
- Node.js 20 + Express
- PostgreSQL + Prisma
- Jest + Supertest (테스트)

## 빌드/테스트 명령어
- 개발 서버: `npm run dev`
- 테스트 전체: `npm test`
- 단일 테스트: `npm test -- --testPathPattern=파일명`
- 린트: `npm run lint`

## 코드 규칙
- require 대신 import/export 사용 (ES Modules)
- 에러 핸들링은 express-async-errors 미들웨어 사용
- 환경변수는 dotenv로 관리, .env 파일은 커밋하지 않음
```

## Step 3. 잘 쓰는 법

Claude Code 창시자 Boris Cherny는 팀에서 CLAUDE.md를 이렇게 운영합니다.

> "Claude가 실수할 때마다 CLAUDE.md에 추가한다. 그러면 다음에는 같은 실수를 하지 않는다."
> "수정을 마치면 'CLAUDE.md에 이 규칙을 추가해'라고 말한다."

**넣어야 할 것**

- Claude가 추측할 수 없는 명령어 (빌드, 테스트, 린트)
- 기본값과 다른 코드 스타일 규칙
- 저장소 관례 (브랜치 네이밍, PR 규칙)
- 프로젝트 고유의 아키텍처 결정
- 비개발 업무의 문서 형식, 톤, 반복 패턴

**빼야 할 것**

- Claude가 코드를 읽으면 알 수 있는 것
- 언어의 표준 관례 (Claude는 이미 알고 있음)
- 자주 바뀌는 정보
- 긴 설명이나 튜토리얼
- "깔끔하게 작성"처럼 자명한 지시

한 줄을 적을 때마다 "이걸 빼면 Claude가 실수할까?"를 따져봅니다. 아니라면 빼는 것이 낫습니다. "깔끔하게 작성"보다 "들여쓰기 2칸, 세미콜론 생략"이 효과적입니다. API 키나 비밀번호는 절대 넣지 않습니다. 프로젝트 CLAUDE.md는 Git에 커밋되므로 노출됩니다.

**파일당 200줄 이내를 권장합니다.** 200줄을 넘으면 별도 파일로 분리합니다. CLAUDE.md 안에서 `@` 구문으로 다른 파일을 가져올 수 있습니다.

```markdown
# 프로젝트 개요
@README.md 참고

# Git 워크플로우
@docs/git-instructions.md

# 개인 설정 (Git에 포함되지 않음)
@~/.claude/my-project-instructions.md
```

## Step 4. 3가지 범위 이해

CLAUDE.md는 세 곳에 둘 수 있고, Claude는 세션 시작 시 세 곳을 모두 읽어서 합칩니다.

| 범위 | 위치 | 용도 | Git 포함 |
|------|------|------|----------|
| 프로젝트 | `./CLAUDE.md` | 팀 공유 규칙 | 포함 (팀원 모두 적용) |
| 사용자 | `~/.claude/CLAUDE.md` | 개인 선호 | 미포함 (나만 적용) |
| 관리 정책 | 시스템 경로 | 조직 전체 정책 | 별도 관리 |

사용자 CLAUDE.md는 내 컴퓨터에만 존재하며 모든 프로젝트에 적용됩니다. Claude에게 바로 만들어달라고 하면 됩니다.

```
~/.claude/CLAUDE.md 파일을 만들어줘. 내용은 이렇게:
- 응답은 한국어로
- 커밋 메시지는 한국어로 작성
- 코드 변경 시 관련 테스트도 함께 수정
- 변수명과 함수명은 영어 사용
```

## Step 5. Auto Memory - Claude가 스스로 적는 메모

CLAUDE.md는 사람이 적는 지시서입니다. Auto Memory는 Claude가 작업하면서 스스로 적는 메모입니다.

| | CLAUDE.md | Auto Memory |
|--|-----------|-------------|
| 누가 작성 | 사람 | Claude |
| 내용 | 지시와 규칙 | 작업 중 발견한 패턴 |
| 범위 | 프로젝트, 사용자, 조직 | 프로젝트별 (내 컴퓨터에만) |
| 용도 | 코딩 표준, 워크플로우, 아키텍처 | 빌드 명령어, 디버깅 패턴, 선호사항 |

예를 들어 Claude가 보고서를 반말로 작성했을 때 "격식체로 써줘"라고 고쳐주면, Claude가 스스로 "이 프로젝트에서는 격식체 사용"이라고 메모합니다. 다음 세션부터는 고쳐줄 필요가 없습니다.

Auto Memory는 기본으로 켜져 있으며, 다음 대화에서 유용할 것이라고 판단될 때만 저장합니다. 세션 시작 시 MEMORY.md의 처음 200줄 또는 25KB만 로드됩니다. `/memory`를 입력하면 저장된 내용을 확인하고, 편집하거나 삭제할 수 있습니다.

## Step 6. 동작 확인

Claude를 새로 실행하고 CLAUDE.md 내용을 질문합니다.

```
이 프로젝트의 작업 규칙이 뭐야?
```

Claude가 CLAUDE.md에 적힌 내용을 인용하며 답하면 정상입니다. 세션을 종료하고 다시 시작해도 CLAUDE.md 내용은 매번 자동으로 읽힙니다.
